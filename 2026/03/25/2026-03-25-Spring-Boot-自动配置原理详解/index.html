⚠️ 本文基于以下版本编写：

  • Spring Framework: 5.3.x / 6.x
  • Java: 8 / 11 / 17

不同版本源码可能存在细微差异，但核心原理一致。

1. 什么是自动配置

1.1 传统 Spring 开发的痛点

在没有 Spring Boot 之前，开发一个 Spring 应用需要：

痛点：

  • 配置繁琐，容易出错
  • 依赖版本冲突难以管理
  • 每个项目都要重复配置

1.2 Spring Boot 的解决方案

Spring Boot 的自动配置（Auto-Configuration）机制：

自动配置做了什么？

  • 根据 classpath 中的依赖，自动配置 Spring 应用
  • 提供合理的默认值，减少配置
  • 同时允许通过配置覆盖默认值

注解 作用

关键：@Import(AutoConfigurationImportSelector.class)

这是自动配置的入口，Spring 会调用 AutoConfigurationImportSelector 加载自动配置类。

4. 条件注解详解

4.1 条件注解的作用

自动配置不是一股脑加载所有配置，而是根据条件选择性加载。

4.2 常用条件注解

注解 条件 示例
@ConditionalOnClass classpath 存在指定类 @ConditionalOnClass(DataSource.class)
@ConditionalOnMissingClass classpath 不存在指定类 -
@ConditionalOnBean 容器中存在指定 Bean @ConditionalOnBean(DataSource.class)
@ConditionalOnMissingBean 容器中不存在指定 Bean @ConditionalOnMissingBean(DataSource.class)
@ConditionalOnProperty 指定属性满足条件 @ConditionalOnProperty(prefix="spring.datasource")
@ConditionalOnWebApplication 是 Web 应用 -
@ConditionalOnResource 存在指定资源 @ConditionalOnResource(resources="classpath:banner.txt")

6.4 自动配置的设计原则

  1. 向后兼容：升级 Spring Boot 版本不应破坏现有配置
  2. 合理默认值：提供开箱即用的体验
  3. 可覆盖：允许用户通过配置或代码覆盖默认行为
  4. 条件加载：只在需要时加载配置
  5. 无侵入：不影响不使用该功能的用户

7. 总结

7.1 核心要点回顾

要点 说明
入口 @EnableAutoConfiguration
配置来源 META-INF/spring.factories 或 META-INF/spring/*.imports
选择器 AutoConfigurationImportSelector
条件过滤 各种 @ConditionalOnXxx 注解
加载时机 Spring 容器 refresh 阶段

7.2 自动配置 vs 手动配置

特性 自动配置 手动配置
开发效率 高 低
灵活性 中等 高
学习成本 需要理解原理 直观
适用场景 快速开发、标准场景 特殊需求、深度定制

7.3 学习建议

  1. 先会用：掌握常用 Starter 的使用
  2. 再理解：阅读自动配置源码
  3. 最后写：尝试编写自定义 Starter
