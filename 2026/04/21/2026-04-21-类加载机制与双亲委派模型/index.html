类加载机制与双亲委派模型

发表于2026-04-21|更新于2026-04-21|JVM性能优化
|字数总计:4.1k|阅读时长:16分钟|阅读量:

版本说明

⚠️ 本文基于以下版本编写：

  • Java: 8 / 11 / 17
  • JVM: HotSpot VM
  • 相关工具: jstat, jmap, jhat, Arthas

前言

类加载机制是 Java 虚拟机的核心组成部分，也是理解 JVM 工作原理的基础。作为 Java 开发者，我们每天都在使用类加载器，却很少深入了解其内部机制。本文将从类加载的过程、类加载器的层次结构、双亲委派模型等方面进行深度剖析，帮助你彻底理解 Java 的类加载机制。

一、类加载的过程

类从被加载到虚拟机内存开始，到卸载出内存为止，它的整个生命周期包括：加载（Loading）、验证（Verification）、准备（Preparation）、解析（Resolution）、初始化（Initialization）、使用（Using）和卸载（Unloading）七个阶段。其中验证、准备、解析三个部分统称为连接（Linking）。

1.1 加载（Loading）

“加载”是”类加载”的一个阶段，主要完成以下三件事：

  1. 通过一个类的全限定名获取定义此类的二进制字节流
  2. 将这个字节流所代表的静态存储结构转化为方法区的运行时数据结构
  3. 在内存中生成一个代表这个类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口

1.2 验证（Verification）

验证是连接阶段的第一步，目的是确保 Class 文件的字节流包含的信息符合当前虚拟机的要求，保证被加载类的正确性，不会危害虚拟机自身的安全。

验证阶段大致会完成以下四个阶段的检验动作：

  • 文件格式验证：验证字节流是否符合 Class 文件格式规范
  • 元数据验证：对字节码描述的信息进行语义分析
  • 字节码验证：通过数据流和控制流分析，确定程序语义是合法的
  • 符号引用验证：对类自身以外的信息进行匹配性校验

1.3 准备（Preparation）

准备阶段是正式为类中定义的变量（静态变量）分配内存并设置类变量初始值的阶段。

1.4 解析（Resolution）

解析阶段是虚拟机将常量池内的符号引用替换为直接引用的过程。

  • 符号引用：以一组符号来描述所引用的目标，引用的目标不一定已经加载到内存中
  • 直接引用：可以直接指向目标的指针、相对偏移量或是一个能间接定位到目标的句柄

1.5 初始化（Initialization）

初始化阶段是类加载过程的最后一步，也是真正开始执行类中定义的 Java 程序代码。初始化阶段就是执行类构造器 <clinit>() 方法的过程。

二、类加载器

类加载器用于实现类的加载动作。Java 虚拟机设计团队把加载动作放到 JVM 外部实现，以便让应用程序自己决定如何去获取所需的类。

2.2 启动类加载器（Bootstrap ClassLoader）

启动类加载器是 JVM 自身的一部分，用 C++ 实现（HotSpot VM），负责加载 JVM 的基础类：

  • 加载 <JAVA_HOME>/lib 目录下的核心类库
  • 加载 -Xbootclasspath 参数指定的路径中的类
  • 加载 rt.jar、resources.jar、charsets.jar 等核心 jar 包

2.3 扩展类加载器（Extension ClassLoader）

扩展类加载器由 sun.misc.Launcher$ExtClassLoader 实现：

  • 加载 <JAVA_HOME>/lib/ext 目录下的扩展类库
  • 加载 java.ext.dirs 系统变量指定的路径中的类

2.4 应用程序类加载器（Application ClassLoader）

应用程序类加载器也叫系统类加载器，由 sun.misc.Launcher$AppClassLoader 实现：

  • 加载用户类路径（ClassPath）上的类库
  • 通过 ClassLoader.getSystemClassLoader() 获取
  • 如果没有自定义类加载器，默认使用此加载器

三、双亲委派模型

3.1 什么是双亲委派模型

双亲委派模型是 Java 类加载器的一种工作机制：如果一个类加载器收到了类加载的请求，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，每一个层次的类加载器都是如此。只有当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。

3.2 双亲委派模型的好处

1. 保证 Java 核心类的安全

双亲委派模型保证了 Java 核心类库的安全，防止用户自定义的类冒充核心类。

2. 避免类的重复加载

同一个类只会被加载一次，保证了类的唯一性。父加载器已经加载过的类，子加载器不会再加载。

四、打破双亲委派模型

双亲委派模型是一个推荐的机制，但不是强制的。在某些场景下，我们需要打破双亲委派模型。

4.1 为什么要打破双亲委派模型

典型场景：SPI（Service Provider Interface）机制

Java 的 SPI 机制允许第三方为某个接口提供实现，如 JDBC。JDBC 的接口定义在 java.sql 包中（由启动类加载器加载），而具体的实现类（如 MySQL 驱动）由第三方提供，通常不在启动类加载器的加载路径中。

问题：DriverManager 类由启动类加载器加载，但它需要加载用户配置的 JDBC 驱动类，这些驱动类不在启动类加载器的路径中。如果严格遵循双亲委派，启动类加载器无法加载这些驱动类。

解决方案：线程上下文类加载器（Thread Context ClassLoader）

六、总结

核心要点回顾

  1. 类加载过程：加载 → 验证 → 准备 → 解析 → 初始化，其中验证、准备、解析统称为连接

  2. 类加载器层次：

    • 启动类加载器（Bootstrap）：加载核心类库
    • 扩展类加载器（Extension）：加载扩展类库
    • 应用程序类加载器（Application）：加载用户类路径

  3. 双亲委派模型：先委派父加载器，父加载器无法加载时才自己加载

  4. 双亲委派的好处：

    • 保证核心类的安全
    • 避免类的重复加载
    • 保证类的唯一性

  5. 打破双亲委派的场景：

    • SPI 机制（线程上下文类加载器）
    • 热部署
    • 模块化系统

最佳实践

  1. 理解类加载顺序：静态初始化块 → 实例初始化块 → 构造器
  2. 避免循环依赖：静态初始化块中不要依赖其他类的初始化
  3. 合理使用自定义类加载器：只在必要时打破双亲委派
  4. 注意类路径配置：确保编译时和运行时类路径一致
  5. 善用诊断工具：Arthas、jstat、jmap 等工具排查问题
