G1 收集器深度剖析

版本说明

⚠️ 本文基于以下版本编写：

  • JDK: 8 / 11 / 17
  • G1 GC: JDK 8u40+ / JDK 11+ / JDK 17+
  • 适用场景: 大内存（4GB+）、低延迟要求的应用

一、G1 收集器概述

G1（Garbage-First）是一款面向服务端应用的垃圾收集器，从 JDK 7u4 开始正式引入，在 JDK 9 中成为默认垃圾收集器。它的设计目标是：

  • 可预测的停顿时间模型：用户可以设置期望的停顿时间上限
  • 高吞吐量：在满足停顿时间要求的同时，尽可能提高吞吐量
  • 大内存友好：适合堆内存 4GB 以上的应用

G1 vs 传统收集器

特性 CMS G1
内存布局 连续分代 Region 分区
回收算法 标记-清除 标记-整理（局部）+ 复制
停顿预测 不可控 可预测
碎片问题 严重 较少
适用场景 中等堆 大堆、低延迟

二、G1 核心设计思想

2.1 Region 分区设计

G1 将堆划分为多个大小相等的独立区域（Region），每个 Region 可以扮演不同的角色：

Region 类型：

  • Eden Region：新生代 Eden 区
  • Survivor Region：新生代 Survivor 区
  • Old Region：老年代区
  • Humongous Region：存放巨型对象（大小 ≥ Region 的 50%）
  • Free Region：空闲区域

Region 大小计算：

  • 最小 1MB，最大 32MB
  • 默认约 2048 个 Region
  • 通过 -XX:G1HeapRegionSize 可手动指定

三、G1 工作流程

3.1 整体阶段

G1 的垃圾回收分为以下几个阶段：

3.2 Young GC（新生代回收）

触发条件： Eden 区满了

执行过程：

  1. 扫描 GC Roots，标记存活对象
  2. 将存活对象复制到 Survivor 或 Old 区
  3. 清空 Eden 和 Survivor 区

3.3 Concurrent Marking（并发标记）

触发条件： 老年代占用达到阈值（默认堆的 45%）

四个子阶段：

阶段 1：初始标记（Initial Mark）

  • STW（Stop-The-World）
  • 仅标记 GC Roots 直接关联的对象
  • 借助 Young GC 完成

阶段 2：并发标记（Concurrent Mark）

  • 并发执行，不阻塞应用
  • 从 GC Roots 遍历整个对象图
  • 使用 SATB（Snapshot-At-The-Beginning）算法

阶段 3：最终标记（Final Mark / Remark）

  • STW
  • 处理 SATB 缓冲区中的剩余引用更新
  • 完成最终标记

阶段 4：清理（Cleanup）

  • STW
  • 统计各 Region 的存活对象数量
  • 将完全空闲的 Region 加入空闲列表
  • 不进行实际的对象移动

3.4 Mixed GC（混合回收）

触发条件： 并发标记完成后

回收范围： 新生代 + 部分老年代

执行策略：

  1. 根据 Garbage First 原则，选择垃圾最多的老年代 Region
  2. 在用户设定的停顿时间内，尽可能多回收
  3. 多次执行，直到老年代垃圾回收完毕

四、G1 关键技术详解

4.1 Remember Set（RSet）与跨代引用

问题： 回收新生代时，老年代可能引用新生代对象

解决方案： 每个 Region 维护一个 RSet，记录”谁引用了我”

RSet 数据结构：

  • PRT（Per Region Table）：哈希表，Key 为引用来源 Region
  • Fine Grain PRT：位图，精确到卡片粒度
  • Coarse Grain PRT：位图，仅记录 Region 粒度

4.2 SATB 算法

SATB（Snapshot-At-The-Beginning）：在并发标记开始时，对堆进行逻辑快照

核心思想：

  • 标记开始时存活的对象，在标记结束时仍视为存活
  • 标记过程中新创建的对象，直接标记为存活
  • 标记过程中引用的删除，通过 Write Barrier 记录

SATB vs 增量更新：

特性 SATB 增量更新（CMS）
记录内容 引用删除时的旧值 引用添加时的对象
精度 粗粒度 细粒度
Remark 开销 低 高
浮动垃圾 较多 较少

4.3 TAMS 指针

TAMS（Top at Mark Start）：标记开始时的堆顶指针

4.4 Humongous Objects（巨型对象）

定义： 大小 ≥ Region Size 的 50% 的对象

存储策略：

  • 单个 Region 放不下，占用多个连续 Region
  • 直接分配在老年代
  • 在并发标记的 Cleanup 阶段回收

优化建议：

  • 避免创建过大的对象
  • 如果必须使用，考虑拆分成小对象
  • 适当增大 Region Size（-XX:G1HeapRegionSize）

五、Full GC 触发条件与调优

5.1 Full GC 触发场景

G1 会尽量避免 Full GC，但在以下情况仍会触发：

  1. Evacuation Failure：复制存活对象时没有足够的空闲 Region
  2. Humongous 分配失败：没有足够的连续 Region 存放巨型对象
  3. To-space Exhausted：复制时目标空间不足
  4. Metadata GC Threshold：元空间不足

七、G1 vs ZGC vs Shenandoah

特性 G1 ZGC Shenandoah
目标停顿 < 200ms < 10ms < 10ms
JDK 版本 7u4+ 15+ (生产就绪) 12+

选择建议：

  • G1：通用场景，堆 4-32GB，停顿要求 100-200ms
  • ZGC：超大堆（TB 级），停顿要求 < 10ms，JDK 15+
  • Shenandoah：大堆，停顿要求 < 10ms，JDK 12+

九、总结

G1 收集器通过 Region 分区 和 可预测停顿模型，在大内存场景下实现了良好的平衡：

优点：

  • 可预测的停顿时间
  • 较少的内存碎片
  • 大堆友好
  • 自适应调优

缺点：

  • 额外的内存占用（RSet 等）
  • 停顿时间仍为毫秒级（不如 ZGC/Shenandoah）
  • 小堆场景不如 Parallel GC

适用场景：

  • 堆内存 4GB 以上
  • 停顿时间要求在 100-500ms
  • 通用服务端应用
