前言

ReentrantLock 是 Java 并发编程中最常用的显式锁，相比 synchronized 关键字，它提供了更丰富的功能：可中断锁、超时锁、公平锁、多条件变量等。本文将从源码角度深入剖析 ReentrantLock 的实现原理，帮助你真正理解它的工作机制。

八、ReentrantLock vs synchronized 对比

特性 ReentrantLock synchronized
实现层面 Java 代码（JUC） JVM 内置
可中断 ✅ lockInterruptibly() ❌
超时获取 ✅ tryLock(timeout) ❌
公平锁 ✅ 可选 ❌ 非公平
多条件变量 ✅ 多个 Condition ❌ 只有一个等待集
性能（JDK 6+） 相近 相近（偏向锁优化后）
代码简洁性 需手动 unlock 自动释放
死锁风险 较低（可超时/中断） 较高

建议： 简单场景优先用 synchronized（更简洁，JVM 可优化）；需要高级特性时用 ReentrantLock。

十、总结

ReentrantLock 的核心设计思路：

  1. AQS 是基础：通过 volatile int state + CAS 操作实现无锁化的状态管理
  2. CLH 队列保证公平性：等待线程排队，按顺序唤醒（公平锁）或允许插队（非公平锁）
  3. LockSupport.park/unpark：底层线程挂起/唤醒机制，比 wait/notify 更灵活
  4. 重入通过 state 计数：同一线程多次加锁，state 累加；解锁时递减，归零才真正释放
  5. Condition 分离等待集：多个条件变量对应多个等待队列，精准唤醒

理解 ReentrantLock 的源码，不仅能帮助你正确使用它，更能让你深入理解 CountDownLatch、Semaphore、CyclicBarrier 等其他 JUC 工具，因为它们都建立在同一套 AQS 框架之上。
