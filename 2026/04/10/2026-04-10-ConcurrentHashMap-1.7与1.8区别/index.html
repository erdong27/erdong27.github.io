ConcurrentHashMap 1.7 与 1.8 区别

ConcurrentHashMap 是 Java 并发编程中最重要的数据结构之一。JDK 1.7 到 1.8 的版本演进中，其实现方式发生了革命性的变化。本文将从源码层面深入分析两个版本的设计差异和性能优化。

版本说明

⚠️ 本文基于以下版本编写：

  • JDK: 1.7.0_80 / 1.8.0_xxx
  • Java: 7 / 8+

JDK 1.8 的 ConcurrentHashMap 实现是理解现代 Java 并发编程的重要基础。

2.4 get 操作无锁设计

为什么 get 不加锁？

  • HashEntry 的 value 和 next 都是 volatile
  • volatile 保证可见性，读取到的是最新值
  • 数组通过 UNSAFE 的 volatile 读取

4.2 put 方法对比

JDK 1.7 put：

  1. 计算 hash，定位 Segment
  2. 尝试获取 Segment 锁（ReentrantLock）
  3. 获取失败则自旋等待
  4. 在 Segment 内部执行 put 操作
  5. 释放锁

JDK 1.8 put：

  1. 计算 hash
  2. 如果数组为空，初始化
  3. 如果目标桶为空，CAS 插入
  4. 如果正在扩容，帮助扩容
  5. 否则 synchronized 锁住头节点，插入链表/红黑树
  6. 检查是否需要树化
  7. 更新计数

对比总结：

特性 JDK 1.7 JDK 1.8
锁粒度 Segment（固定 16 个） 桶级别（动态）
锁类型 ReentrantLock synchronized + CAS
最大并发 16（受限于 Segment 数量） 数组长度（更多）
树化 无 链表 → 红黑树

5. 扩容机制对比

5.1 JDK 1.7 扩容（Segment 级别）

特点：

  • 每个 Segment 独立扩容
  • 不影响其他 Segment
  • 扩容时持有 Segment 锁

5.2 JDK 1.8 扩容（多线程协作）

JDK 1.8 扩容特点：

  1. 多线程协作：多个线程共同参与扩容
  2. 任务分配：每个线程处理一段连续的桶
  3. ForwardingNode：标记已迁移的桶，读请求转发到新数组
  4. 帮助扩容：写线程检测到扩容时会协助迁移

8. 总结

8.1 核心差异对比

特性 JDK 1.7 JDK 1.8
数据结构 Segment + HashEntry 数组 + 链表 Node 数组 + 链表/红黑树
锁粒度 Segment（默认 16 个） 桶级别（数组长度）
锁类型 ReentrantLock synchronized + CAS
并发度 固定（Segment 数量） 动态（数组长度）
get 操作 无锁（volatile） 无锁（volatile）
put 操作 锁 Segment CAS + synchronized
扩容 Segment 独立扩容 多线程协作扩容
size 遍历 Segment LongAdder 思想
树化 不支持 链表 → 红黑树
内存占用 较高（Segment 数组） 较低（无 Segment）

8.2 JDK 1.8 的主要改进

  1. 锁粒度更细：从 Segment 级别降低到桶级别，并发度大幅提升
  2. CAS 优化：插入空桶时无需加锁，减少锁竞争
  3. 红黑树结构：解决链表过长时的性能问题
  4. 多线程扩容：充分利用多核 CPU，扩容效率更高
  5. 内存优化：移除 Segment 数组，减少内存占用

8.3 学习建议

  1. 理解设计思想：从分段锁到 CAS + synchronized 的演进
  2. 关注关键点：volatile 可见性、CAS 原理、红黑树转换
  3. 动手实践：写 Demo 验证各种场景下的行为
  4. 阅读源码：结合注释理解每个设计决策的原因
