JVM内存模型与垃圾回收算法

发表于2026-04-15|更新于2026-04-15|JavaJVM
|字数总计:3.7k|阅读时长:14分钟|阅读量:

版本说明

⚠️ 本文基于以下版本编写：

  • JDK: 8 / 11 / 17
  • JVM: HotSpot
  • GC: 基于 G1 和 ZGC 讲解

前言

Java 之所以能够”一次编写，到处运行”，很大程度上得益于 JVM（Java Virtual Machine）的抽象。与 C/C++ 需要手动管理内存不同，Java 通过自动垃圾回收（Garbage Collection，简称 GC）让开发者从内存管理的繁琐中解放出来。但理解 JVM 内存模型和 GC 原理，是写出高性能 Java 应用的基础。

本文将深入剖析 JVM 内存模型的结构、各个区域的作用，以及主流垃圾回收算法的核心思想。

一、JVM 内存模型（JMM）结构

JVM 在运行期间，会把所管理的内存划分为若干个不同的数据区域。根据《Java 虚拟机规范 SE 8》，整体结构如下：

1.1 线程私有区域

这些区域随线程的创建而创建，随线程的销毁而回收，不存在垃圾回收问题。

程序计数器（Program Counter Register）

  • 当前线程所执行的字节码行号指示器
  • 如果线程正在执行 Java 方法，记录的是正在执行的虚拟机字节码指令地址
  • 如果是 Native 方法，则为空（Undefined）
  • 唯一一块不会发生 OutOfMemoryError 的区域

虚拟机栈（VM Stack）

  • 每个线程拥有一个私有的虚拟机栈
  • 存储局部变量表、操作数栈、动态链接、方法返回地址等信息
  • 每个方法调用对应一个栈帧（Stack Frame）

常见异常：

  • StackOverflowError：栈深度超出允许范围（递归调用没有正确终止）
  • OutOfMemoryError：无法申请足够的内存（线程数量过多）

本地方法栈（Native Method Stack）

  • 为 JVM 使用到的 Native 方法（如 JNI 调用）服务
  • HotSpot 将虚拟机栈和本地方法栈合并实现

1.2 线程共享区域

这些区域被所有线程共享，是垃圾回收的主要关注点。

堆（Heap）

  • JVM 管理内存中最大的一块
  • 几乎所有对象实例和数组都在这里分配
  • 是垃圾回收的主要管理区域，因此也称为 “GC 堆”

2.3 线程安全问题

由于多个线程可能同时在堆上分配对象，需要解决并发问题。

TLAB（Thread Local Allocation Buffer）

  • 为每个线程预分配一块 Eden 区内存
  • 线程优先在 TLAB 中分配，避免同步开销

三、垃圾回收算法

3.1 判断对象是否存活的算法

引用计数算法（Reference Counting）

每个对象有一个引用计数器，引用+1，失效-1。

问题：无法处理循环引用！

可达性分析算法（Reachability Analysis）

从 GC Roots 出发，向下搜索，搜索路径称为”引用链”。

GC Roots 包括：

  • 虚拟机栈中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI 引用的对象
  • JVM 内部引用（ClassLoader、异常对象等）
  • 同步锁持有的对象
  • 反映 JVM 内部情况的 JMXBean、Callback 等

3.2 垃圾回收算法

标记-清除算法（Mark-Sweep）

缺点：

  • 效率不稳定
  • 产生内存碎片

复制算法（Copying）

将内存分为两块，每次只使用一块。存活对象复制到另一块，然后整块清除。

优点：无碎片，效率高
缺点：可用内存减半

实际应用：新生代的 Eden 区和 Survivor 区

标记-整理算法（Mark-Compact）

优点：无碎片，内存利用率高
缺点：移动成本高，需要更新引用

分代收集算法（Generational Collection）

当前主流 JVM 采用的策略，根据对象生命周期特点分区处理：

Minor GC（Minor GC/Young GC）：

  • 发生在新生代
  • 对象优先在 Eden 分配
  • Survivor 区满时触发
  • 频率高，停顿时间短

Major GC/Full GC：

  • 发生在老年代
  • Major GC 通常伴随 Minor GC
  • 频率低，但停顿时间长

4.2 新生代收集器

Serial 收集器

  • 单线程工作
  • 停止所有用户线程（Stop The World）
  • 简单高效，Client 模式默认

ParNew 收集器

  • Serial 的多线程版本
  • Server 模式首选的新生代收集器
  • 可与 CMS 配合使用

Parallel Scavenge 收集器

  • 关注吞吐量（Throughput）
  • 自适应调节策略
  • 适合后台运算任务

4.3 老年代收集器

Serial Old 收集器

  • Serial 的老年代版本
  • 使用标记-整理算法
  • 作为 CMS 的后备收集器

Parallel Old 收集器

  • Parallel Scavenge 的老年代版本
  • 使用标记-整理算法
  • JDK 1.6+ 可用

CMS 收集器（Concurrent Mark Sweep）

  • 以最短停顿时间为目标
  • 并发收集，低停顿
  • 使用标记-清除算法

缺点：

  • 对 CPU 资源敏感
  • 无法处理浮动垃圾
  • 产生内存碎片

4.4 G1 收集器（Garbage First）

JDK 9+ 默认收集器。将堆划分为多个大小相等的 Region（1MB~32MB），跟踪各 Region 的垃圾价值（回收获得的空间和回收所需时间），优先回收价值最大的 Region。

特点：

  • 并行与并发
  • 分代收集
  • 空间整合（Region 间采用复制算法）
  • 可预测停顿

八、总结

本文深入探讨了 JVM 内存模型和垃圾回收算法的核心知识：

核心要点：

  1. 内存区域划分：线程私有（程序计数器、虚拟机栈、本地方法栈）和线程共享（堆、方法区）

  2. 对象分配：优先在 Eden 区分配，使用 TLAB 或 CAS 解决并发问题

  3. 死亡对象判定：可达性分析算法，以 GC Roots 为起点

  4. 垃圾回收算法：标记-清除、复制、标记-整理，以及综合运用的分代收集

  5. 垃圾收集器：从 Serial 到 G1、ZGC，各有适用场景

  6. 性能调优：监控 → 分析 → 调优 → 验证的闭环流程

理解这些底层原理，不仅能帮助我们写出更高效的代码，还能在系统出现性能问题时快速定位和解决。
