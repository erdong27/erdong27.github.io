线程池拒绝策略自定义

版本说明

⚠️ 本文基于以下版本编写：

  • Java: 8 / 11 / 17
  • Spring Boot: 2.7.x / 3.x（示例代码兼容）

前言

在上一篇文章中，我们深入探讨了 ThreadPoolExecutor 的核心参数调优策略。然而，即使我们精心调校了线程池的各项参数，也无法完全避免任务被拒绝的场景。当线程池已满且队列已满时，RejectedExecutionHandler 就会发挥作用。

本文将全面解析 Java 线程池的四种内置拒绝策略，并手把手教你如何自定义拒绝策略来应对生产环境中的各种复杂场景。

一、拒绝策略的核心机制

1.1 RejectedExecutionHandler 接口

Java 提供了一个简洁而强大的接口来定义任务被拒绝时的行为：

当线程池无法接受新任务时，会调用 rejectedExecution() 方法。方法接收两个关键参数：

  • r：被拒绝的任务
  • executor：执行任务的线程池实例

1.2 触发时机

拒绝策略在以下两种情况会被触发：

  1. 线程池已关闭：shutdown() 或 shutdownNow() 被调用后
  2. 线程池容量饱和：线程池中的线程数达到 maximumPoolSize，且工作队列已满

二、四种内置拒绝策略详解

2.1 AbortPolicy（默认策略）

AbortPolicy 是线程池的默认拒绝策略，抛出 RejectedExecutionException 异常。

特点：

  • 直接抛出异常，调用方需要自行处理
  • 适用于对任务丢失敏感的场景
  • 便于快速发现问题

2.2 CallerRunsPolicy（调用者执行策略）

CallerRunsPolicy 由调用线程自己执行被拒绝的任务。

特点：

  • 保证任务不丢失
  • 由于调用线程也参与执行，会对调用方产生压力
  • 实际上是 一种背压机制，可以减缓任务提交速度
  • 适用于轻量级调用方，或作为降级策略

使用示例：

实战场景：防止上游服务被高并发压垮

2.3 DiscardPolicy（丢弃策略）

DiscardPolicy 静默丢弃被拒绝的任务，不抛出任何异常。

特点：

  • 任务被无声无息地丢弃
  • 不推荐在对任务完整性有要求的场景使用
  • 适用于对实时性要求高、可容忍数据丢失的场景（如监控数据）

使用示例：

2.4 DiscardOldestPolicy（丢弃最旧策略）

DiscardOldestPolicy 丢弃队列中最旧（最早）的任务，然后尝试重新提交新任务。

特点：

  • 优先保证新任务的执行
  • 丢弃旧任务可能导致数据不一致
  • 适用于优先级任务场景（新的高优先级任务应该优先执行）

三、自定义拒绝策略实战

3.1 为什么需要自定义拒绝策略？

内置的四种策略各有局限：

策略 优点 缺点
AbortPolicy 明确失败 需要异常处理
CallerRunsPolicy 背压机制 阻塞调用方
DiscardPolicy 无感知 任务丢失
DiscardOldestPolicy 保证新任务 丢弃重要旧任务

生产环境中，我们往往需要：

  1. 记录详细日志：便于问题排查
  2. 持久化任务：如写入数据库或消息队列等待后续处理
  3. 发送告警通知：第一时间感知系统异常
  4. 动态扩容：或触发其他补偿机制

五、最佳实践总结

5.1 选择合适的拒绝策略

场景 推荐策略
核心交易系统 AbortPolicy + 完善的异常处理
高并发 API CallerRunsPolicy 作为背压机制
监控/日志采集 DiscardPolicy 容忍数据丢失
任务调度系统 自定义持久化策略
实时数据处理 DiscardOldestPolicy 或优先级策略

5.2 通用最佳实践

  1. 始终监控拒绝策略触发次数

  2. 设置合理的队列容量

    • 队列过大：任务等待时间过长，可能超时
    • 队列过小：频繁触发拒绝策略

  3. 优雅关闭线程池

  4. 拒绝策略应该幂等且快速

    • 不要在拒绝策略中执行耗时操作
    • 避免抛出新的异常（除非有明确的错误处理）

结语

线程池的拒绝策略是系统稳定性的最后一道防线。选择合适的拒绝策略，需要综合考虑业务场景、数据一致性要求、系统负载等因素。

本文介绍的自定义策略仅为抛砖引玉，实际生产中可能还需要：

  • 与分布式追踪系统集成（如 SkyWalking、Zipkin）
  • 支持动态调整策略
  • 与配置中心联动实现热更新

希望本文能帮助你在实际项目中构建更加健壮的异步任务处理系统。
