一、为什么需要分布式事务？

在微服务架构下，一个业务操作可能涉及多个服务、多个数据库。传统的本地事务（ACID）无法保证跨服务的数据一致性。

1.1 典型场景

以电商下单为例，一个订单操作涉及：

  • 订单服务：创建订单记录
  • 库存服务：扣减商品库存
  • 支付服务：处理支付信息
  • 积分服务：增加用户积分

如果库存扣减成功但订单创建失败，就会出现数据不一致的问题。

1.2 CAP 理论与 BASE 理论

CAP 定理指出，分布式系统无法同时满足：

  • Consistency（一致性）：所有节点同时看到相同的数据
  • Availability（可用性）：每个请求都能得到响应
  • Partition Tolerance（分区容错性）：网络分区时系统仍能运行

在分布式系统中，P 是必须保证的，因此只能在 C 和 A 之间权衡。

BASE 理论是对 CAP 的补充：

  • Basically Available（基本可用）：允许损失部分可用性
  • Soft State（软状态）：允许中间状态存在
  • Eventually Consistent（最终一致性）：保证最终数据一致

三、TCC 模式详解

3.1 TCC 原理

TCC（Try-Confirm-Cancel）是一种补偿型事务模式，将业务逻辑分为三个阶段：

3.2 以转账为例说明 TCC

假设 A 账户向 B 账户转账 100 元：

Try 阶段：

  • 检查 A 账户余额是否充足
  • 冻结 A 账户 100 元（余额减少，冻结金额增加）
  • 预增 B 账户 100 元（预增金额字段增加）

Confirm 阶段：

  • 扣减 A 账户冻结金额
  • B 账户预增金额转为实际余额

Cancel 阶段：

  • 解冻 A 账户冻结金额
  • 清除 B 账户预增金额

3.4 TCC 关键问题与解决方案

问题一：空回滚

场景：Try 未执行成功，Cancel 却被调用

解决方案：记录事务ID，Cancel 时检查 Try 是否执行

问题二：悬挂

场景：Cancel 先于 Try 执行（网络延迟），Try 无法执行

解决方案：Try 执行前检查是否已回滚

问题三：幂等性

场景：Confirm/Cancel 被重复调用

解决方案：状态机 + 事务ID

四、Seata 框架实战

4.1 Seata 简介

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里开源的分布式事务解决方案，提供 AT、TCC、SAGA、XA 四种事务模式。

4.2 AT 模式原理

AT（Automatic Transaction）模式是 Seata 的核心模式，特点是无侵入、自动补偿。

工作原理：

  1. 一阶段：执行业务 SQL，记录前后镜像到 UNDO_LOG
  2. 二阶段提交：异步删除 UNDO_LOG
  3. 二阶段回滚：根据 UNDO_LOG 反向生成补偿 SQL

六、总结

6.1 核心要点回顾

要点 说明
CAP 理论 分布式系统无法同时满足 C、A、P
BASE 理论 基本可用、软状态、最终一致
TCC 模式 Try-Confirm-Cancel，高并发场景首选
Seata AT 无侵入、自动补偿，开发效率高
幂等性 防止重复执行的关键
空回滚 Try 未执行但 Cancel 被调用
悬挂 Cancel 先于 Try 执行

6.2 实战经验

  1. 优先选择 AT 模式：开发成本低，维护简单
  2. 高并发场景选 TCC：性能要求高时
  3. 做好幂等设计：所有分布式事务都需要
  4. 合理设置超时：避免长时间占用资源
  5. 监控与告警：及时发现和处理异常
  6. 做好回滚预案：极端情况下的兜底方案
