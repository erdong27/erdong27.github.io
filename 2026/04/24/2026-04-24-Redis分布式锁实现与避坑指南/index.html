Redis 分布式锁实现与避坑指南

版本说明

⚠️ 本文基于以下版本编写：

  • Redis: 6.x / 7.x
  • Redisson: 3.x
  • Spring Boot: 2.7.x / 3.x
  • Java: 8 / 11 / 17

前言

在分布式系统中，多个服务实例同时访问共享资源时，本地锁（如 synchronized、ReentrantLock）就失效了。分布式锁应运而生，而 Redis 凭借其高性能和丰富的特性，成为实现分布式锁的热门选择。

然而，Redis 分布式锁的实现看似简单，实则暗藏诸多陷阱。本文将从最基础的实现出发，逐步演进到生产级方案，并深入分析各种边界场景和解决方案。

一、分布式锁的核心要求

在讨论实现之前，先明确分布式锁必须满足的核心特性：

1.1 基本特性

特性 说明
互斥性 任意时刻，只有一个客户端能持有锁
防死锁 锁必须能自动释放，避免持有者崩溃导致死锁
可重入 同一线程可多次获取同一把锁
高可用 锁服务要有足够的容错能力
高性能 获取/释放锁的延迟要低

1.2 额外考虑

  • 锁归属识别：释放锁时必须验证是本人持有，防止误删
  • 锁续期：业务执行时间超过锁过期时间怎么办？
  • 集群一致性：主从切换时锁会不会丢失？

问题一：非原子操作

setnx 和 expire 是两条命令，如果 setnx 成功后程序崩溃，锁没有过期时间，造成死锁。

问题二：锁误删

A 线程获取锁后执行超时，锁过期自动释放。B 线程获取锁，A 线程执行完毕释放锁，实际释放的是 B 的锁。

3.3 为什么必须用 Lua 脚本？

解锁需要两步：

  1. 判断锁是否属于自己（GET）
  2. 删除锁（DEL）

如果不用 Lua，可能出现：

  • 线程 A 执行 GET 发现有锁
  • 锁刚好过期，线程 B 获取锁
  • 线程 A 执行 DEL，删除了 B 的锁

Lua 脚本在 Redis 中原子执行，避免竞态条件。

四、生产级方案：Redisson

4.1 为什么选择 Redisson？

自己实现的锁还存在以下问题：

问题 说明
锁续期 业务执行时间 > 锁过期时间，锁自动释放
可重入 同一线程多次获取锁需要计数支持
集群容错 主从切换时锁可能丢失
等待队列 获取锁失败的线程如何等待重试？

Redisson 提供了完整的解决方案。

4.3 Redisson 核心特性

Watch Dog（看门狗）自动续期

Watch Dog 机制：

  • 锁默认 30 秒过期
  • 后台线程每 10 秒（1/3 过期时间）检查锁是否还被持有
  • 如果持有，自动续期到 30 秒
  • 业务完成后主动 unlock()，Watch Dog 停止续期

五、集群环境下的陷阱

5.1 主从切换导致锁丢失

场景：

  1. 客户端 A 在 Master 获取锁成功
  2. Master 还没同步给 Slave 就宕机
  3. Slave 升级为新 Master
  4. 客户端 B 在新 Master 获取锁成功
  5. A 和 B 同时持有锁！

核心思想：向多个独立的 Redis 节点申请锁，只有多数节点（N/2 + 1）成功才算获取成功。

5.3 Redlock 的争议

分布式系统专家 Martin Kleppmann 对 Redlock 提出质疑：

  • 依赖系统时钟，时钟跳跃可能导致问题
  • GC 暂停可能导致锁过期判断错误

实际建议：

  • 对安全性要求极高的场景（如金融），考虑 ZooKeeper
  • 一般业务场景，单 Redis + 合理过期时间足够

七、分布式锁选型对比

特性 Redis ZooKeeper etcd
性能 高 中 中
可靠性 中（需 Redlock） 高 高
实现复杂度 简单 复杂 中等
CAP 取向 AP CP CP
锁续期 手动/WATCHDOG 临时节点自动续期 Lease
使用场景 高并发、可容忍短暂不一致 强一致性要求 云原生场景

八、总结

Redis 分布式锁从简单到完善，需要考虑：

  1. 原子性：SET NX EX 或 Lua 脚本
  2. 归属识别：UUID 标识 + Lua 校验删除
  3. 自动续期：Watch Dog 机制
  4. 可重入：Redisson Hash 结构
  5. 集群容错：Redlock 或切换 ZooKeeper

生产环境推荐直接使用 Redisson，它已经解决了大部分坑。但作为开发者，理解其背后的原理和潜在问题同样重要——这样在遇到问题时才能快速定位和解决。
