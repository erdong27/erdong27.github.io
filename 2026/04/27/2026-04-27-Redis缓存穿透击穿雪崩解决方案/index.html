Redis 缓存穿透/击穿/雪崩解决方案

发表于2026-04-27|更新于2026-04-27|Java分布式
|字数总计:4.1k|阅读时长:17分钟|阅读量:

版本说明

⚠️ 本文基于以下版本编写：

  • Redis: 6.x / 7.x
  • Spring Boot: 2.7.x / 3.x
  • Spring Data Redis: 2.7.x / 3.x
  • Redisson: 3.x
  • Java: 8 / 11 / 17

前言

缓存是提升系统性能的利器，但如果使用不当，可能引发三大经典问题：缓存穿透、缓存击穿、缓存雪崩。这三个问题在高并发场景下可能直接压垮数据库，是每个后端开发者必须掌握的知识点。

本文将深入分析这三个问题的成因、危害，并给出生产可用的解决方案，附带完整代码示例。

一、缓存穿透（Cache Penetration）

1.1 什么是缓存穿透？

缓存穿透是指查询一个根本不存在的数据，缓存层和数据库层都没有命中，请求每次都会”穿透”缓存直达数据库。

危害：恶意攻击者可以构造大量不存在的 key，绕过缓存层，直接打垮数据库。

1.2 解决方案一：缓存空值

最简单的方案：查询数据库返回空时，也将空值写入缓存，设置较短的过期时间。

优点：实现简单，效果立竿见影。

缺点：

  • 占用缓存空间（大量空值 key）
  • 数据一致性问题：空值缓存期间，数据库新增了该数据，缓存仍返回空

1.3 解决方案二：布隆过滤器（推荐）

布隆过滤器（Bloom Filter）是一种空间效率极高的概率型数据结构，可以快速判断一个元素一定不存在或可能存在。

原理：

  • 初始化时，将所有合法的 key 写入布隆过滤器
  • 请求到来时，先经过布隆过滤器判断
  • 如果布隆过滤器说”不存在”，直接返回，不查缓存和数据库
  • 如果布隆过滤器说”可能存在”，再走正常查询流程

使用 Redisson 实现布隆过滤器

布隆过滤器的特性：

  • 空间效率高：100 万个元素，误判率 1%，只需约 1.2MB
  • 查询速度快：O(k)，k 为哈希函数个数
  • 不支持删除：元素一旦加入无法删除（可用 Counting Bloom Filter）
  • 存在误判：可能将不存在的元素判断为存在（假阳性），但不会将存在的元素判断为不存在

二、缓存击穿（Cache Breakdown）

2.1 什么是缓存击穿？

缓存击穿是指一个热点 key 在某一时刻过期，恰好此时有大量并发请求访问该 key，所有请求都会穿透缓存直达数据库，造成数据库瞬间压力剧增。

与缓存穿透的区别：

  • 穿透：key 根本不存在
  • 击穿：key 存在，但在某一时刻过期

2.2 解决方案一：互斥锁（Mutex Lock）

只允许一个线程去查数据库并重建缓存，其他线程等待。

优点：数据一致性强，缓存中的数据始终是最新的。

缺点：等待期间性能下降，存在死锁风险（需设置锁过期时间）。

2.3 解决方案二：逻辑过期（推荐热点数据）

不设置 key 的真实过期时间，而是在 value 中存储一个逻辑过期时间。查询时判断逻辑时间是否过期，过期则异步更新，当前请求返回旧数据。

优点：高并发下性能好，不会阻塞请求。

缺点：数据存在短暂不一致（返回旧数据），不适合强一致性场景。

2.4 解决方案三：永不过期（适合极热点数据）

对于极热点数据（如首页 Banner、热门商品），可以设置永不过期，通过定时任务主动更新缓存。

三、缓存雪崩（Cache Avalanche）

3.1 什么是缓存雪崩？

缓存雪崩是指在某一时刻，大量缓存 key 同时过期，或者 Redis 服务宕机，导致大量请求直接打到数据库，造成数据库压力骤增甚至崩溃。

两种场景：

  1. 大量 key 同时过期：批量写入缓存时设置了相同的过期时间
  2. Redis 宕机：缓存层完全失效

3.2 解决方案一：过期时间随机化

在基础过期时间上加随机偏移，避免大量 key 同时过期。

3.3 解决方案二：多级缓存

构建多级缓存体系，即使 Redis 宕机，本地缓存仍可提供服务。

五、三大问题对比总结

问题 触发条件 核心解决方案 适用场景
缓存穿透 查询不存在的 key 布隆过滤器 / 缓存空值 防恶意攻击
缓存击穿 热点 key 过期 互斥锁 / 逻辑过期 热点数据保护
缓存雪崩 大量 key 同时过期 / Redis 宕机 随机过期时间 / 多级缓存 / 高可用 大规模缓存场景

方案选型建议

缓存穿透：

  • 数据量可控 → 缓存空值（简单）
  • 数据量大、有恶意攻击风险 → 布隆过滤器

缓存击穿：

  • 数据一致性要求高 → 互斥锁
  • 性能优先、可接受短暂不一致 → 逻辑过期
  • 极热点数据 → 永不过期 + 定时刷新

缓存雪崩：

  • 必做：过期时间随机化
  • 推荐：多级缓存（本地 + Redis）
  • 生产必备：Redis 高可用（哨兵/集群）
  • 保底：熔断降级 + 限流

六、总结

缓存三大问题是 Redis 使用中的经典难题，理解其本质才能选择合适的解决方案：

  • 穿透：数据不存在，用布隆过滤器在入口拦截
  • 击穿：热点数据过期，用锁或逻辑过期控制并发重建
  • 雪崩：大规模失效，用随机化 + 多级缓存 + 高可用兜底

在实际项目中，这三个问题往往需要组合应对。建议将常用逻辑封装为工具类，统一管理缓存策略，降低业务代码的复杂度。
